# Building RocketRide Nodes (v2)

A node name in, a shipped, conventions-compliant node out. The **mechanical skeleton is generated**
(`builder nodes:new`) so this skill carries only **judgment + gates**. v1 split this across 13 skills
that restated `services.json` rules in ~16 files and hardcoded base-class line numbers that rot; v2
folds the mechanics into the generator (CI-verified) and the durable rules into one skill + the
always-on AGENTS.md block.

## Discipline (also in AGENTS.md — restated because this skill can load standalone)

The four gates are **hard stops**. **Waiting means ENDING YOUR TURN.** A dismissed dialog, an
unanswered question, or a headless session is a STOP, never an approval. Never "proceed with
defaults"; never write "scope was confirmed at Gate X" unless a human answered. An unbuilt node
costs nothing; an unapproved one wastes the build and the review.

Read all reference material via `git show origin/develop:<path>` — **never trust the local working
tree** (it may be a stale branch). Resolve counts/paths/exemplars live; do not memorize them.

## Phases

0. **Locate + staleness.** Find the checkout; `git fetch origin develop`. Confine all writes to
   `nodes/src/nodes/<name>/`, `nodes/test/`, `examples/`, `docs/`. Anything in `packages/` (core) is
   a **written suggestion for a human**, never a unilateral edit.
1. **Research.** Light dual-direction for a clone of an existing archetype; deep/multi-agent only for
   novel nodes. Goal: the vendor's full operation surface + the right archetype + 1–2 exemplars.
2. **GATE A — archetype + scope.** Present: detected archetype + the `--archetype` value you'll pass
   to `nodes:new`; the 1–2 exemplars (read live); the **complete operation menu** (every operation by
   name, partitioned proposed / deferred / out-of-scope, closed by "Menu complete: N operations,
   verified against <source>"); per-operation archetype check (a tool-shaped op among data-flow ops →
   surface the dual `classType` option, even if MVP excludes it); and what the node will NOT be.
   **If the node — or a variant/preset — already exists, Gate A becomes "here's the evidence, what's
   needed beyond it?"** (zero code may be the answer).
   > PR #1063 built an MCP-client clone when the ask was a Supabase **DB** node; the right answer was
   > a `db_postgres` variant. Wrong archetype = total rework. Confirm even when you're sure.
3. **Scaffold + mockup.** Run `builder nodes:new --archetype <X> --vendor <v> --title <T>` (add
   `--dry-run` first to show the file list as the mockup). The generator emits the structure,
   `services.json` skeleton, base-class wiring, test stub, README markers, and `node-ops.json`.
   Present the blast radius (should be the node dir + test only).
4. **GATE B — approve mockup + blast radius.**
5. **Plan.** Phases sized to complexity (provider clone 1–2; agent node 5+). Each phase ends with the
   per-phase e2e audit, not just "tests pass". For a 1–2-phase clone, B and C may be one combined
   gate — say so explicitly. Gate A is never combined.
6. **GATE C — approve the plan.**
7. **Build, phase by phase.** Fill the generator's TODOs — the **judgment** layer (operations,
   request/response shaping, model lists, config fields). Tests written *inside* each phase. Run
   `builder nodes:test-contracts` continuously; it must stay green.
8. **Final e2e + self-review** of the diff. Confirm `node-ops.json` lists the real operations and the
   README prose around the markers is written (the markers themselves are regenerated by
   `nodes:docs-generate`).
9. **GATE D — ask how to close out** (full close-out / PR / draft PR / local branch). Never assume.
   Branch `feat/RR-<issue>-...`, base `develop`, conventional commits.

## Archetype dispatch (the collapsed v1 archetype skills)

`nodes:new` owns the file layout, base classes, and `services.json` shape per archetype. This table
is the **judgment** that the generator can't make for you:

| Archetype | `--archetype` | Exemplar (read live) | Judgment notes |
|---|---|---|---|
| Tool (wrap a vendor API) | `tool` | `tool_v0` | One `@tool_function` per operation. **Raise on error — never return `{success:False}`** (tavily/firecrawl ship that grandfathered style). `lanes:{}`. |
| LLM provider | `llm` | `llm_anthropic` | `IInstance(LLMBase)` is ~100% boilerplate; the work is `IGlobal` validation + the `<vendor>.py` Chat class + model profiles. `lanes:{questions:[answers]}`. |
| Data-flow processor | `processor` | `anomaly_detector` | Lane handlers (`writeText`/`writeDocuments`), not `@tool_function`. Put real logic in a helper module. Pick `classType` from real lane types. |
| Relational DB | `database` | `db_postgres` | Dual `classType:["database","tool"]`. Implement only `_connection_params`/`_build_connection_url` (IGlobal) + `_db_display_name`/`_db_dialect` (IInstance); the base does the rest. |
| Hosted flavor of an existing engine (Supabase, Neon, RDS…) | `database --variant-of <node>` | `db_postgres/services.supabase.json` | **Not a new node** — a `services.<flavor>.json` variant reusing the same code path. |
| embedding / vector-store / ingress / agent | (extend the generator) | `embedding_*`, `pinecone`, `webhook`, `agent_*` | Not yet templated; scaffold by hand from the live exemplar following the same anatomy. Adding a template is the natural next generator PR. |

## Staying current (deterministic, no LLM)

`node-ops.json` records `supported_operations` + `openapi_url`/`changelog_url`. `node-ops-diff`
(a plain script / scheduled Action) diffs it against the vendor's live OpenAPI and opens a **draft**
PR when the vendor adds an operation — draft-only, never auto-merged, zero model calls. The Gate-A
menu rule still governs whether a detected operation belongs in scope.

## Red flags

| Thought | Reality |
|---|---|
| "Obviously a tool node, skip Gate A" | #1063 was "obvious" and built the wrong archetype entirely. |
| "I'll hand-write the files, faster than the generator" | Re-deriving boilerplate is how base-class paths rot and `experimental` gets dropped. Scaffold, then fill judgment. |
| "Dismissed dialog / nobody answered, proceed" | Dismissed = unanswered = STOP. Deliver the gate brief and end your turn. |
| "Local clone is fine, no need to fetch" | Read `origin/develop`; develop moves fast and branches go stale. |
| "Researched that op but it doesn't fit MVP, leave it off the menu" | Every op goes on the menu (deferred is fine); the count line makes omissions visible. |
| "Small fix in `packages/ai`, I'll just do it" | Core changes are written suggestions for a human, always. |
